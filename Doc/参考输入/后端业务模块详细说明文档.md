## 1. 系统架构总览

### 1.1 技术栈

- **运行时**: Electron 主进程 (Node.js)
- **语言**: TypeScript
- **通信**: Electron IPC (ipcMain/ipcRenderer)
- **持久化**: JSON 文件 (原子写入)
- **外部库**: Python (DAQ-P-1604 驱动、五孔插值)、C# (辅助采集程序)、koffi (FFI 绑定)

### 2.3 AppRuntime

**文件**: `src/main/bootstrap/AppRuntime.ts`

AppRuntime 是应用运行时容器，聚合所有核心服务实例并提供依赖注入能力。

**核心组件**:
- `DeviceRegistry` / `MotionRegistry` — 设备/运动控制器注册表
- `DeviceRuntimeFacade` / `MotionRuntimeFacade` — 运行时门面
- `AcquisitionRuntime` — 采集运行时
- `SystemEventBus` — 系统事件总线
- `ResourceLockService` — 资源锁服务
- `HealthMonitor` — 健康监控

**关键方法**:
- `buildIpcDeps(): IpcDeps` — 构造 IPC 依赖注入对象
- `markReady()` — 标记运行时就绪，发布 healthy 状态事件
- `clearMainServices()` — 清空所有服务引用（窗口关闭时调用）

### 2.4 窗口关闭处理

- 主窗口关闭: `acquisitionHub.dispose()` → `dataStorageService.stopRecording()` → `appRuntime.clearMainServices()`
- `window-all-closed`: 非 macOS 平台退出应用

## 3. 设备管理模块

### 3.1 DeviceManager

**文件**: `src/main/DeviceManager.ts`

DeviceManager 是整个 DAQ 系统的设备管理核心，负责设备生命周期管理、IPC 通信桥接、配置同步与采集控制。

#### 3.1.1 核心属性

| 属性 | 类型 | 说明 |
|------|------|------|
| `store` | `DeviceStoreManager` | 设备配置持久化存储 |
| `drivers` | `Map<string, BaseDevice>` | 设备ID → 驱动实例映射 |
| `instances` | `Map<string, DeviceInstance>` | 设备ID → 运行时实例映射 |
| `dataSink` | `((data: DataPayload) => void) \| null` | 数据接收回调（外部注入） |
| `scanService` | `DeviceScanServiceLike` | 设备扫描服务 |
| `scanInFlight` | `Promise<DiscoveredDevice[]> \| null` | 防止并发扫描的锁 |

#### 3.1.2 核心方法

| 方法 | 签名 | 功能说明 |
|------|------|----------|
| `setDataSink` | `(sink) => void` | 设置数据下沉回调，数据从设备驱动流向 AcquisitionHub 和 DataStorageService |
| `scanDevices` | `() => Promise<DiscoveredDevice[]>` | 扫描设备（防并发，单次飞行锁） |
| `init` | `async () => Promise<void>` | 初始化实例同步 + 自动连接 |
| `getProfiles` | `() => DeviceProfile[]` | 获取所有设备配置 |
| `connectAll` | `async () => Promise<void>` | 并发连接所有设备 |
| `disconnectAll` | `async () => Promise<void>` | 并发断开所有设备 |
| `startAcquisitionAll` | `async () => Promise<number>` | 批量启动采集，返回成功数 |
| `stopAcquisitionAll` | `async () => Promise<void>` | 批量停止采集 |
| `connect` | `async (id: string) => Promise<boolean>` | 连接设备（创建驱动、注册回调、同步单位） |
| `disconnect` | `async (id: string) => Promise<void>` | 断开设备 |

#### 3.1.3 IPC 通道

| IPC 通道 | 功能 | 参数 | 返回值 |
|----------|------|------|--------|
| `device:getProfiles` | 获取所有配置 | - | `DeviceProfile[]` |
| `device:getInstances` | 获取所有实例 | - | `DeviceInstance[]` |
| `device:getStatusAll` | 获取所有状态 | - | `DeviceStatus[]` |
| `device:getCapabilities` | 获取能力 | - | `DeviceCapabilities` |
| `device:upsertProfile` | 更新/插入配置 | `DeviceProfile` | `DeviceProfile` |
| `device:deleteProfile` | 删除配置 | `id: string` | `void` |
| `device:connect` | 连接设备 | `id: string` | `boolean` |
| `device:disconnect` | 断开设备 | `id: string` | `void` |
| `device:startAcquisition` | 启动采集 | `id: string` | `boolean` |
| `device:stopAcquisition` | 停止采集 | `id: string` | `void` |
| `device:scan` | 扫描设备 | - | `DiscoveredDevice[]` |
| `device:setUnit` | 设置压力单位 | `id, unit` | `void` |
| `device:getDaqT1603Config` | 获取T1603硬件配置 | `id` | `DaqT1603HardwareConfig` |
| `device:applyDaqT1603Config` | 应用T1603硬件配置 | `id, config` | `void` |

#### 3.1.4 设备连接流程

```
connect(id)
  │
  ├─ 1. 从 store 获取 profile
  ├─ 2. sanitizeProfile(profile) — 校验/清洗配置
  ├─ 3. DriverFactory.createDriver(profile) — 创建驱动实例
  ├─ 4. driver.onData(callback) — 注册数据回调
  │     └─ callback: data → dataSink(data) → AcquisitionHub + DataStorageService
  ├─ 5. driver.onConfigSynced = callback — 注册配置同步回调
  ├─ 6. driver.connect() — 建立硬件连接
  ├─ 7. [DAQ-P-1604] syncUnitFromDevice() — 同步压力单位
  ├─ 8. 更新 instance.status = 'Connected'
  └─ 9. notifyStatusChange() — 广播状态变更
```

#### 3.1.5 配置热更新

当 `device:upsertProfile` 被调用时：
1. 校验并保存新配置
2. 如果设备已连接且正在采集，先停止采集
3. 如果设备驱动支持 `updateChannels`，调用热更新（不重连）
4. 如果设备正在采集，重新启动采集

#### 3.1.6 错误处理

- `connect()` 失败: 设置 `instance.status = 'Error'` 和 `lastError`
- 采集重启失败: 设置 `instance.acquiring = false`
- 单位同步失败: 非阻塞，仅打印警告

## 4. 设备驱动层

### 4.1 BaseDevice (抽象基类)

**文件**: `src/main/drivers/BaseDevice.ts`

所有设备驱动的基类，定义连接/断开/采集的抽象接口，提供重连、数据回调、配置更新等通用能力。

#### 抽象方法（子类必须实现）

| 方法 | 签名 | 说明 |
|------|------|------|
| `connect` | `abstract () => Promise<boolean>` | 建立连接 |
| `disconnect` | `abstract () => Promise<void>` | 断开连接 |
| `startAcquisition` | `abstract () => Promise<boolean>` | 启动采集 |
| `stopAcquisition` | `abstract () => Promise<void>` | 停止采集 |

#### 重连机制

指数退避算法：
```
delay = min(RECONNECT_BASE_DELAY_MS * 2^reconnectAttempts, RECONNECT_MAX_DELAY_MS)
```
- `RECONNECT_BASE_DELAY_MS` = 1000ms
- `RECONNECT_MAX_DELAY_MS` = 30000ms
- `MAX_RECONNECT_ATTEMPTS` = 5

重连成功后重置计数器，达到最大次数后停止重连。

#### 配置同步回调

```typescript
onConfigSynced: (channels: ChannelConfig[], hardwareConfig: unknown) => void
```

设备连接后或硬件配置变更时触发，用于将硬件实际配置同步到本地存储。

### 4.2 DAQP1604Device (DAQ-P-1604 压力采集仪)

**文件**: `src/main/drivers/DAQP1604Device.ts`

基于 TCP/IP 连接的 16 通道压力采集设备驱动，使用自定义协议（非 Modbus），支持数据流模式和轮询模式。

#### 4.2.1 连接参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `host` | `192.168.3.101` | 设备 IP 地址 |
| `port` | `9000` | TCP 端口 |
| `streamId` | `1` | 数据流 ID |

#### 4.2.2 通道结构

18 通道：CH1~CH16（压力）+ CH17（大气压）+ CH18（大气温度）

#### 4.2.3 通信协议

**帧格式**: `| lenHi(1B) | lenLo(1B) | payload...(len-2 字节) |`

2 字节大端长度前缀，前 2 字节为长度本身，后续为有效载荷。

**命令列表**:

| 命令 | 功能 | 响应 |
|------|------|------|
| `w1601` | 启用 2 字节长度前缀模式 | ASCII 确认 |
| `c 00 <params>` | 配置数据流参数 | ASCII 确认 |
| `c 05 <params>` | 配置数据流返回内容 | ASCII 确认 |
| `c 01` | 启动数据流 | 二进制数据帧 |
| `c 02` | 停止数据流 | ASCII 确认 |
| `rFFFF0` | 轮询 16 通道压力（备用调试） | ASCII 数据 |

**数据流帧结构**:
```
| 头(5B) | CH1(4B, float32 BE) | CH2 | ... | CH16 | CH17(大气压) | CH18(温度) |
```
共 5 + 18×4 = 77 字节

#### 4.2.4 连接流程

```
connect()
  ├─ 创建 TCP Socket (net.Socket)
  ├─ 连接到 host:port
  ├─ 发送 w1601 启用长度前缀模式
  ├─ 注册 Socket 事件:
  │   ├─ 'data' → handleSocketData() — 2字节长度前缀拆包
  │   ├─ 'error' → handleDisconnect() — 触发自动重连
  │   └─ 'close' → handleDisconnect()
  └─ 返回 true
```

#### 4.2.5 采集流程

```
startAcquisition()
  ├─ configureStream(periodMs) — 配置数据流参数
  │   ├─ c 00: 内部时钟 / 大端二进制 / 连续采集
  │   └─ c 05: 返回压力+大气压+温度
  ├─ 发送 c 01 启动数据流
  └─ 返回 true

数据接收:
  handleSocketData(chunk)
    ├─ 追加到 recvBuffer
    ├─ 按 2 字节长度前缀拆包
    ├─ 判断帧类型 (ASCII / 二进制)
    ├─ 二进制帧 → handleStreamFrame()
    │   ├─ 解析 5 字节头 + 18 个 float32(BE)
    │   ├─ 映射到通道数据
    │   └─ 调用 onDataCallback(DataPayload)
    └─ ASCII 帧 → 命令响应队列
```

#### 4.2.6 扩展接口

| 方法 | 签名 | 功能 |
|------|------|------|
| `sendRawCommand` | `async (command: string, timeoutMs?: number) => Promise<string>` | 发送原始命令并等待响应 |
| `updateChannels` | `(channels: ChannelConfig[]) => void` | 热更新通道配置 |

### 4.3 DAQT1603Device (DAQ-T-1603 热电偶温度采集仪)

**文件**: `src/main/drivers/DAQT1603Device.ts`

支持 TCP 和串口双传输模式的 16 通道热电偶温度采集设备驱动。

#### 4.3.1 连接参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `host` | `192.168.1.7` | TCP 地址 |
| `port` | `9000` | TCP 端口 |
| `serialPortPath` | - | 串口路径（如 COM3） |
| `baudRate` | `460800` | 波特率 |

#### 4.3.2 通道结构

16 通道热电偶温度，每通道可配置热电偶类型（K/B/E/J/T/S/N/R/C/WRE325/WRE526/WRE520）。

#### 4.3.3 TCP 模式协议

**命令列表**:

| 命令 | 功能 | 响应 |
|------|------|------|
| `@f0 <mask> 2` | 启动连续采集 | 64 字节二进制数据帧 |
| `@f3 <types>` | 设置热电偶类型 | ASCII 确认 |
| `@e3` | 读取热电偶类型 | ASCII 数据 |
| `@fd <param>` | 读取硬件参数 | ASCII 数据 |
| `@fe <param> <value>` | 设置硬件参数 | ASCII 确认 |

**TCP 二进制数据帧**:
```
64 字节 = 16 通道 × 4 字节 (Little-Endian float32)
通道顺序: CH15 → CH0 (反转)
```

**响应分发策略**:
1. 64 字节二进制帧优先识别
2. ASCII 响应按换行符切分
3. 固定长度响应按期望长度切分
4. 可变长度响应按静默窗口(30ms)收尾

#### 4.3.4 串口模式协议

**命令**:
- 启动采集: `55 AA 03 F0 00 00` (6 字节)
- 停止采集: `55 AA 03 F1 00 00` (6 字节)

**串口数据帧**:
```
| 帧头(2B) | 长度(1B) | 帧计数(1B) | 反帧计数(1B) | 状态(1B) | 版本(2B) | 温度(32B) | 备用(4B) | 校验(1B) |
共 46 字节
温度: 16 通道 × 2 字节 (Big-Endian int16), 0.1℃/LSB
```

#### 4.3.5 硬件配置同步

连接后自动执行 `syncHardwareConfig()`:
1. `readThermocoupleTypes()` — 读取热电偶类型 (@e3)
2. `readHardwareConfig()` — 逐个读取硬件参数 (@fd 系列)
3. `updateChannelsFromHardware()` — 更新本地通道配置

#### 4.3.6 扩展接口

| 方法 | 签名 | 功能 |
|------|------|------|
| `getHardwareConfig` | `async () => Promise<DaqT1603HardwareConfig \| null>` | 获取当前硬件配置 |
| `applyHardwareConfig` | `async (config) => Promise<void>` | 应用硬件配置到设备 |
| `updateChannels` | `(channels) => void` | 热更新通道配置 |

### 4.4 SimulatedDevice (模拟设备)

**文件**: `src/main/drivers/SimulatedDevice.ts`

用于开发调试的模拟设备驱动，生成正弦波+随机噪声的模拟数据。

**数据生成公式**:
```typescript
fullChannels[i] = Math.sin(Date.now() / 1000 + i) * 10 + (Math.random() - 0.5)
```
- 每个通道: 正弦波(幅度10) + 随机噪声(±0.5)
- 按 `samplingRate` 频率定时生成
- 只发送 `enabled` 通道的数据


#### 7.3.3 编码器补偿

B140MotionController 实现了完整的编码器补偿状态机：

```
waiting-stop → settling → checking → compensating → succeeded / failed / cancelled
```

**补偿流程**:
1. `moveTo`/`moveBy` 执行后，若轴启用了编码器补偿，创建 `PendingCompensationRequest`
2. 在 `getAllAxisStatus` 轮询时，检测到运动停止后激活补偿任务
3. 补偿循环:
   - 读取编码器实际位置
   - 计算与目标位置的误差
   - 若误差在容差内 → 同步寄存器并标记成功
   - 否则 → 发送修正运动并继续等待
4. 支持最大循环次数、超时、最小步长等保护参数

**补偿配置** (AxisEncoderCompensationConfig):

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `enabled` | `false` | 是否启用 |
| `tolerance` | `0.01` | 补偿容差（工程单位） |
| `maxCycles` | `3` | 最大补偿次数 |
| `settleMs` | `100` | 稳定等待时间(ms) |
| `minStep` | `0` | 最小补偿步长 |
| `timeoutMs` | `5000` | 补偿超时(ms) |

#### 7.3.4 串行命令锁

B140Driver 使用 `runExclusive(fn)` 保证同一时刻仅一条命令在传输，避免命令交错。

### 8.4 五孔探针校准 (FiveHoleCalibration)

**文件**: `src/main/calibration/FiveHoleCalibration.ts`

#### 通道匹配策略

role + 名称双重匹配:
1. 优先按 `ProbeChannelRole` 匹配（如 `fiveHole.p1`）
2. 回退按通道名称匹配（如 `P1`）

#### 校准系数计算

```typescript
// 压力系数
Kα = (P4 - P5) / Δ
Kβ = (P3 - P1) / Δ
其中 Δ = P2 - (P1 + P3 + P4 + P5) / 4

// 总压/静压系数
CPT = f(Kα, Kβ)  — 从校准表插值
CPS = g(Kα, Kβ)  — 从校准表插值
```

#### 实时系数推送

采集每个样本后，以 100ms 节流间隔推送 `calibration:realtime` 事件到前端，包含原始数据和计算系数。

### 8.5 三孔探针校准 (ThreeHoleCalibration)

**文件**: `src/main/calibration/ThreeHoleCalibration.ts`

#### 校准系数计算

```typescript
K  = (P1 - P3) / (P1 - 2*P2 + P3)   // 方向系数
Cv = (P1 - P3) / (2 * P2 - P1 - P3) // 速度系数
Cp = (P2 - P_total) / (P1 - P2)      // 总压系数
```

### 8.6 总压探针校准 (TotalPressureCalibration)

**文件**: `src/main/calibration/TotalPressureCalibration.ts`

仅移动 α 轴（单轴校准）。

#### 校准系数计算

```typescript
// 表压转绝压
Pt_tunnel = P_tunnel_total + P_atm    // 风洞总压（绝压）
Ps_tunnel = P_tunnel_static + P_atm   // 风洞静压（绝压）

// 总压系数
CPT = (Pt_probe - Ps_tunnel) / (Pt_tunnel - Ps_tunnel)

// 测量误差
error = (1 - CPT) * 100  // 百分比

// 马赫数
Ma = sqrt(5 * ((Pt_tunnel / Ps_tunnel)^(2/7) - 1))
```

### 8.7 总温探针校准 (TotalTemperatureCalibration)

**文件**: `src/main/calibration/TotalTemperatureCalibration.ts`

手动模式校准，不自动移动，由操作员手动调节风洞工况。

#### 特殊逻辑

1. **5Hz 马赫数监控**: 实时计算并推送当前马赫数
2. **温度稳定性检测**: 基于标准差判据

   - 标准差 < `maxStdDev`
   - 采样间隔 `sampleInterval`
3. **恢复系数计算**:

```typescript
r = (T_probe - T_static) / (T_total - T_static)
```

#### 校准流程

```
  │     └─ 5Hz 监控: 计算当前 Ma，推送 calibration:realtime
  │
  ├─ 2. 等待温度稳定
  │     └─ 连续采样，检查标准差 < maxStdDev
  │
  ├─ 3. 采集数据
  │     └─ 被校探针温度 + 标准探针温度 + 压力参数
  │
  ├─ 4. 计算恢复系数
  │
  └─ 5. 推送进度事件
```

### 9.4 OptimizedRealtimeInterpolator

**文件**: `src/main/services/traversal/OptimizedRealtimeInterpolator.ts`

优化的实时插值服务，集成缓存、节流和异步计算三大优化策略。

#### 优化策略

1. **缓存优先**: 先尝试从 `InterpolationCache` 查找，命中则直接返回
2. **异步计算**: 启用异步时防并发复用（同一时刻仅一个计算在进行）
3. **节流推送**: 结果推送到 `RealtimeUpdateThrottler`，由节流器决定何时通知 UI

#### 配置选项

| 选项 | 默认值 | 说明 |
|------|--------|------|
| `updateFrequencyHz` | `10` | UI 刷新频率(Hz) |
| `cacheSize` | `100` | 缓存大小 |
| `cacheTolerance` | `10` | 压力匹配容差(Pa) |
| `enableCache` | `true` | 是否启用缓存 |
| `enableAsync` | `true` | 是否启用异步计算 |

#### 计算流程

```
calculateRealtime(pressures)
  │
  ├─ 1. 尝试缓存命中 → 命中则直接返回
  ├─ 2. 执行插值计算 (异步/同步)
  ├─ 3. 存储到缓存
  └─ 4. 推送到节流器 → 按频率通知 UI
```

### 9.5 InterpolationCache

**文件**: `src/main/services/traversal/InterpolationCache.ts`

插值结果缓存器，使用 LRU 策略管理缓存，支持容差匹配。

**量化策略**: `Math.round(value / tolerance) * tolerance`，将连续压力值离散化以减少缓存条目。

**LRU 淘汰**: 综合考虑 `hitCount`（命中次数）和 `timestamp`（时间戳），优先淘汰低命中+最旧的条目。

### 9.6 ScanValveAdapter

**文件**: `src/main/services/traversal/ScanValveAdapter.ts`

**核心流程**:
1. 设备数据到来 → 更新缓冲区

### 10.2 PrbInterpolator (单 PRB 文件插值器)

**文件**: `src/main/services/interpolation/PrbInterpolator.ts`

实现完整的 9 区域五孔探针插值算法。

#### PRB 文件格式

- 首行: `13 13`（网格维度，-30° 到 +30°，步长 5°）
- 后续 169 行: 每行 6 列 `ka kb cpt cps alpha beta`

#### 9 区域插值算法

1. **计算压力系数**:
   ```
   Ka = (P4 - P5) / Δ
   Kb = (P3 - P1) / Δ
   Δ = P2 - (P1 + P3 + P4 + P5) / 4
   ```

2. **区域 1-8 解析** (边界区域):
   - 区域 1/3/5/7: 四个角点区域，Ka/Kb 超出校准表范围时夹紧到角点
   - 区域 2/6: β 边界区域，沿 β=±30° 边界线性插值 α
   - 区域 4/8: α 边界区域，沿 α=±30° 边界线性插值 β

3. **区域 9 解析** (内部区域):
   - 在预计算的 12×12 个四边形单元中查找包含点(Ka,Kb)的单元
   - 通过到对边的距离比插值 α 和 β

4. **输出插值**:
   - 对 CPT/CPS 进行双线性插值
   - 总压: `Pt = P2 - CPT * Δ`
   - 静压: `Ps = Pavg - CPS * Δ`

5. **气动计算**:
   ```
   V = sqrt(2 * |Pt - Ps| * R * T / (Patm + Ps))
   Ma = sqrt(5 * |((Pt + Patm) / (Ps + Patm))^(2/7) - 1|)
   ```

### 10.3 MultiPrbInterpolator (多 PRB 文件插值器)

**文件**: `src/main/services/interpolation/MultiPrbInterpolator.ts`

支持不同马赫数的多个校准文件，根据实时 Ma 值自动选择或插值。

#### 插值模式

| 模式 | 说明 |
|------|------|
| `'nearest'` | 选择 Ma 最接近的 PRB 文件直接计算 |
| `'linear'` | 在两个相邻 Ma 的 PRB 结果之间线性加权 |

#### 计算流程

```
calculate(pressures)
  │
  ├─ 1. 用中间 Ma 的 PRB 计算初始 Ma 估计值
  │
  ├─ 2. 根据插值模式:
  │     ├─ nearest → 选择最接近的 PRB 文件直接计算
  │     └─ linear → 找到 Ma 所在区间
  │           ├─ 分别用两个 PRB 计算结果
  │           └─ 对各字段线性加权
  │
  └─ 3. 超出 Ma 范围时退化为最近邻
```

### 18.5 关键设计模式

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| 策略模式 | BaseDevice / BaseMotionController | 多种驱动实现统一接口 |
| 观察者模式 | SystemEventBus / AcquisitionHub | 发布/订阅事件通知 |
| 工厂模式 | DriverFactory | 根据类型创建驱动实例 |
| 门面模式 | DeviceRuntimeFacade / MotionRuntimeFacade | 简化运行时操作 |
| 注册表模式 | DeviceRegistry / MotionRegistry | 解耦服务实例管理 |
| 生产者-消费者 | WriteQueue + DeviceFileWriter | 异步批量写入 |
| 依赖注入 | IpcDeps + AppRuntime.buildIpcDeps() | IPC 层依赖注入 |
| 状态机 | B140 编码器补偿 | waiting-stop → settling → checking → compensating → succeeded/failed |
| 指数退避 | BaseDevice.handleDisconnect | 自动重连策略 |
| LRU 缓存 | InterpolationCache | 插值结果缓存淘汰 |
| 节流 | RealtimeUpdateThrottler | UI 更新频率控制 |
